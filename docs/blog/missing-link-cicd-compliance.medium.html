Every modern engineering team runs security scanners. Snyk checks dependencies. SonarQube flags code smells. GitHub Advanced Security opens Dependabot alerts. Trivy scans containers.

These tools are good at what they do. They find CVEs, flag misconfigurations, and surface exposed secrets.

But here's what none of them do: tell you which SOC 2 control is affected. Which NIST 800-53 requirement just failed. Which CMMC practice you need to document remediation for.

That gap costs GRC teams 40+ hours every quarter.

The Compliance Gap Nobody Talks About

Here's what "vulnerability management" actually looks like at most companies:

  1. CI/CD scanner finds 200 vulnerabilities
  2. Security engineer exports results to CSV
  3. GRC analyst opens Excel
  4. They manually map each CVE to NIST, SOC 2, or CMMC controls
  5. They build POAM entries by hand
  6. Repeat every 90 days

This isn't a tooling problem. It's an integration problem.

The scanning tools and the compliance tools don't talk to each other. Vulnerability data lives in one system. Control mappings live in a spreadsheet. The POAM lives in another spreadsheet. The audit trail lives in someone's email.

Compliance becomes a quarterly fire drill instead of a continuous process.

Why Scanner Vendors Haven't Fixed This

Scanner vendors optimize for detection. They compete on coverage — how many CVEs in the database, how many languages supported, how fast the scan runs.

Compliance frameworks are a different world entirely. They define controls, not vulnerabilities.

A single CVE might map to three different NIST 800-53 controls. A misconfigured Terraform resource might violate both SOC 2 CC6.1 (logical access) and CMMC AC.L2-3.1.1 (authorized access control).

Building the bridge between "CVE-2024-38816 found in Spring Framework" and "this affects NIST SI-2, SOC 2 CC7.1, and CMMC SI.L2-3.14.1" requires a mapping layer that neither scanner vendors nor GRC platforms have built.

The Chain That Makes It Work

Every vulnerability has a CWE (Common Weakness Enumeration) classification:

  • CWE-89 = SQL Injection
  • CWE-79 = Cross-Site Scripting
  • CWE-798 = Hard-Coded Credentials

NIST maintains mappings from CWEs to 800-53 controls. And every major compliance framework maps back to NIST 800-53.

The chain:

Vulnerability → CWE → NIST 800-53 → SOC 2 / CMMC / FedRAMP / ASVS / SSDF

This chain is well-documented. It's not proprietary. But nobody had automated it inside the CI/CD pipeline.

So we did.

One Command. Four Scanners. Six Frameworks.

We built Pipeline Compliance Scanner to close this gap. It's a free CLI that runs four scanners in parallel and maps every finding through the full compliance chain.

What gets scanned:

  • Dependencies (SBOM) — package-lock.json, yarn.lock, pnpm-lock.yaml, requirements.txt, Cargo.lock, go.sum, Gemfile.lock, pom.xml, build.gradle. Generates a CycloneDX 1.5 SBOM and cross-references every package against known advisories.
  • Secrets — 30+ credential patterns (AWS, Azure, GCP, GitHub, Slack, Stripe, private keys, DB connection strings) plus Shannon entropy detection for unrecognized secret formats.
  • Infrastructure as Code — 29 rules across Terraform, CloudFormation, Kubernetes manifests, and Dockerfiles. Checks CIS benchmarks: public S3 buckets, missing encryption, overprivileged IAM, exposed ports, disabled logging, root containers.
  • API Security NEW — Static analysis of Next.js API routes. Catches missing authentication, missing RBAC enforcement, missing CSRF protection on mutations, missing org scoping (tenant isolation gaps), mass assignment, missing rate limiting, and missing input validation. Maps to OWASP Top 10 and CWE IDs.

Auto-Triage: Cutting Through the Noise

Raw scanner output is noisy. A secrets scanner that flags API_KEY=your-key-here in a README template isn't finding a real secret. A dependency match on "express" when the advisory targets "express-validator" is wasted time.

Every finding gets one of three automatic verdicts:

  • TRUE_POSITIVE — Real vulnerability. Maps to compliance controls. Act on it.
  • FALSE_POSITIVE — Auto-dismissed. Test fixtures, .env.example files, sample data, regex literals, charset constants, substring-matched packages.
  • NEEDS_REVIEW — Ambiguous. Gitignored secrets, unknown package versions, edge cases that need a human decision.

Here's the part that matters: the exit code only counts TRUE_POSITIVE findings. Your CI pipeline won't break because of a false positive in a test file. And auto-dismissed findings still show in the output — tagged [FP] with a reason — so nothing is hidden.

Your team focuses on findings that actually affect your compliance posture instead of chasing noise.

Four Output Formats NEW

Scanner output is only useful if it fits into your workflow. The CLI supports four formats:

# Colored terminal output with compliance impact summary (default)
crp-scan --format table

# Machine-readable JSON — pipe to jq, feed to your tools, upload to CVERiskPilot
crp-scan --format json | jq '.complianceImpact'

# GitHub-flavored markdown tables — ready for PR comments
crp-scan --format markdown >> SECURITY.md

# SARIF 2.1.0 — integrates with VS Code, GitHub Code Scanning
crp-scan --format sarif > results.sarif

SARIF is the standard interchange format for static analysis tools. If your team uses GitHub Code Scanning, VS Code Problem Matcher, or any SARIF-compatible viewer, your compliance scan results show up alongside your existing SAST findings.

Built for Pipelines, Not Quarterly Meetings

Drop it into GitHub Actions:

name: Compliance Scan
on: [push, pull_request]
jobs:
  scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - name: Install dependencies
        run: npm ci
      - name: Compliance scan
        run: |
          npx @cveriskpilot/scan@latest \
            --preset enterprise \
            --api-routes \
            --ci \
            --fail-on critical

--ci outputs structured JSON with color disabled. --fail-on sets your severity gate. --api-routes enables the API security scanner for Next.js projects. Exit codes:

  • 0 — No TRUE_POSITIVE findings at or above your threshold
  • 1 — Findings detected (only true positives count)
  • 2 — Scanner error (config issue, missing files)

Pick your compliance scope with presets:

  • startup — SOC 2 + ASVS (getting enterprise-ready)
  • enterprise — NIST 800-53 + SOC 2 + ASVS + SSDF
  • federal — NIST 800-53 + FedRAMP + SSDF
  • defense — NIST 800-53 + CMMC + SSDF
  • devsecops — ASVS + SSDF
  • all — All 6 frameworks

Or pick exactly the frameworks you need: crp-scan --frameworks soc2,cmmc,asvs

Aliases work — soc2, soc, soc-2 all resolve to the same thing. Run crp-scan --list-frameworks to see everything.

Advanced Filtering NEW

Real projects have known accepted risks, noisy CWEs, and directories you don't control. The CLI handles all of it:

# Only show HIGH and CRITICAL findings
crp-scan --severity HIGH --preset startup

# Exclude accepted risks from your risk register
crp-scan --exclude-cwe CWE-79,CWE-89 --preset enterprise

# Skip generated code and vendor directories
crp-scan --exclude "generated/**" --exclude "vendor/**"

# Run only one scanner
crp-scan --deps-only    # Dependencies
crp-scan --secrets-only # Secrets
crp-scan --iac-only     # Infrastructure as Code
crp-scan --api-routes   # API routes (Next.js)

What the CLI Does and Doesn't Do

Clarity matters. The CLI is the scanner. The platform is where compliance becomes continuous.

The CLI does:

  • Scan dependencies, secrets, IaC, and API routes
  • Map every finding to 6 compliance frameworks (135 controls)
  • Auto-triage findings (TRUE_POSITIVE / FALSE_POSITIVE / NEEDS_REVIEW)
  • Output in 4 formats (table, JSON, markdown, SARIF)
  • Run offline, locally, with zero dependencies

The CLI does NOT:

  • Generate POAMs — that's a platform feature
  • AI-powered deep triage — the CLI uses heuristics; AI triage is on the platform
  • Discover zero-day vulnerabilities — it maps known CVEs and CWEs
  • Certify your SOC 2 compliance — it maps findings to SOC 2 controls
  • Monitor continuously — it runs on-demand; continuous monitoring is a platform feature

Use It as a Library

@cveriskpilot/scan also works as a Node.js library. Embed scanning into your own tools, custom CI scripts, or internal dashboards:

import { scanDependencies, scanSecrets, scanIaC, formatOutput } from '@cveriskpilot/scan';

const deps = await scanDependencies('/path/to/project');
const secrets = await scanSecrets('/path/to/project');
const iac = await scanIaC('/path/to/project');

const allFindings = [...deps, ...secrets, ...iac];
const output = formatOutput(allFindings, {
  format: 'json',
  frameworks: ['soc2-type2', 'owasp-asvs']
});
